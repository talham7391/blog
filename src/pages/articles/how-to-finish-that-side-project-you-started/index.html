How to Finish That Side Project You Started

I’ve had a lot of success with the approach I took developing my last side project (a multiplayer card game called Estimation). In this article, I will outline my approach and hopefully you can apply these principles in your next project to get it across the finish line.

Quick disclaimer - this article talks about how to finish your projects, not why you should finish them. Why you should finish your projects deserves its own article.

Imagine we start working on a side project that will require 10,000 lines of code to complete. We get to 3,000 lines and realize we’re starting to get bored but decide to grind it out. Then we get to 5,000 lines, but no further. From here on out our time is spent on YouTube videos (of people shoving their face into bread) until we’re hit with another jolt of inspiration and decide to start a new project. And so the vicious cycle continues. The solution to this problem is quite simple really… stop working on projects that require 10,000 lines of code to be considered complete!

In formal terms, we’re talking about project scope. Often times, people underestimate the time and development required to get a project to the finish line. This is dangerous because either you’ll spend many months finishing a project or abandon it before completion. Both outcomes less than ideal. So how should one approach project scope?

Define an MVP (Minimum Viable Product). An MVP should only have all the necessary features to provide value. In my case, it meant my game shouldn’t have animations, fancy art, match making, in-game purchases, etc… I decided my MVP would allow 4 friends to come together, create a game, and play several rounds of Estimation together.

The great part about this approach is that once you’ve completed your MVP, you have the option of walking away and you’ll have a project which you can use/demo. You did what you set out to do, and you can now move on. However, if you want to start developing other features, you still can. Once you’re bored, you can walk away and not worry about time wasted on a half finished project.

How I Applied This Approach to My Project

It’s hard to judge how much time a project will take to finish, especially when you’ve never done anything similar to it before. I’ve developed REST APIs before but this was my first project using sockets and multiplayer communication. I couldn’t find a good proxy to estimate how much time this would take. So this is how I approached my MVP strategy:

I vertically split the project into 3 smaller projects.

Lets see what each sub-project is responsible for.

Estimation Logic

  • Facilitates estimation gameplay.
  • Asserts game logic on player input.

https://github.com/talham7391/estimation

Web Service

  • Allows players to create games.
  • Facilitates multiplayer gameplay for clients.

https://github.com/talham7391/estimationserver

Game Client

  • A desktop and mobile app.

https://github.com/talham7391/estimation-godot

Each sub-project requires completion of the previous one. Inherently, this creates the following dependency graph. Game Logic <- Web Service <- Game Client. This approach/structure works amazingly well for many different reasons:

  • I can stop after completing any subproject and I will have created something of value.
  • It greatly reduces coupling in my implementation.
  • I can change my game logic by editing the Estimation library. I don’t have to touch the Web Server or Client Code and everything will just work.
  • I can add new clients (web app, command line, etc...) very easily.
  • If I decide to create an AI I can have a library which can run simulations - something that would’ve been very difficult and would’ve required a lot of changes had I baked my game logic into the web server.
  • It made my code very testable.

Conclusion

I’ve only touched the surface of why project scoping is so important. The article is getting long but I can still write more paragraphs outlining the beneficial side effects it has. Project scoping is a skill like any other. The more you practice it, the better you’ll get. Keep focusing on it and eventually you’ll be a project scope pro! Have a vision for your project, break it down into smaller steps, and take the first one with an MVP.
